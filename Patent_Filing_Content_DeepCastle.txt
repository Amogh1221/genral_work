4. PROBLEM BEING SOLVED BY THIS INVENTION
────────────────────────────────────────────────────────────────────────

PROBLEM STATEMENT:

State-of-the-art chess engines such as Stockfish 18 have demonstrated
superhuman chess strength, but their evaluation functions (NNUE networks)
are trained exclusively by large development teams using billions of
positions across weeks of distributed computing. The complete pipeline
— from raw training data generation through neural network design,
quantization, binary export, C++ integration, and full-stack web
deployment — has never been documented or made accessible as a unified,
reproducible end-to-end system.

Specific challenges this invention addresses:

1. KNOWLEDGE GAP: The full NNUE training pipeline (data preparation →
   feature engineering → neural architecture design → quantization-aware
   training → binary export → engine integration) is technically
   demanding, fragmented across multiple repositories, and rarely
   documented end-to-end outside the Stockfish core team.

2. EVALUATION QUALITY: Training an NNUE on human game databases (e.g.,
   Lichess) causes loss to stagnate — empirically confirmed in this
   project, where training on unfiltered Lichess evaluation data caused
   training loss to stagnate at 0.029 for 16+ epochs with zero
   improvement. The correct approach (engine self-play quiet-position
   data with multi-PV filtering) was not previously accessible as a
   reproducible reference implementation.

3. DUAL-NETWORK SWITCHING: Modern engines require two simultaneous NNUE
   networks (BigNet for balanced positions, SmallNet for materially
   imbalanced positions) with adaptive runtime switching. Implementing
   this from scratch alongside a custom-trained brain requires deep
   integration across training, quantization, and C++ inference code.

4. DEPLOYMENT BARRIER: Chess engines are traditionally desktop-only
   executables with no web interface. Making a custom-trained engine
   accessible via a modern web application with real-time evaluation,
   game analysis, multiplayer, and Chess960 support has not been
   done as an open, documented, deployable system.

5. RESOURCE CONSTRAINTS: Training NNUE networks typically requires
   industrial-scale GPU clusters. This invention demonstrates that
   effective NNUE networks can be trained on consumer-grade hardware
   (NVIDIA RTX 3060 / RTX 4050 Laptop) within practical time frames.

ALTERNATIVE SOLUTIONS AVAILABLE AND THEIR LIMITATIONS:
- Stockfish + standard weights: Uses pre-built weights, not custom-trained;
  no reproducible pipeline for training custom brains.
- Leela Chess Zero (LCZero): Uses Monte Carlo Tree Search + deep
  residual networks requiring GPU inference and TPU-class hardware;
  incompatible with classical alpha-beta search frameworks.
- AlphaZero: Requires specialised tensor hardware; not deployable on
  commodity CPUs; not open-source.
- Generic NNUE training tutorials: Scattered, incomplete, and not
  integrated into a production-ready web-deployed system.

5. BRIEF DESCRIPTION OF THE INVENTION AND HOW IT WORKS
────────────────────────────────────────────────────────────────────────

PURPOSE:
DeepCastle is an end-to-end chess engine ecosystem that trains two custom
NNUE (Efficiently Updatable Neural Network) evaluation functions from
scratch using Stockfish self-play data, integrates them into a
Stockfish-derived C++ alpha-beta search engine, and deploys the complete
system as a full-stack web application accessible to any browser user.

HOW IT WORKS (7-Stage Pipeline):

STAGE 1 — DATA GENERATION:
Training data is generated via Stockfish's gensfen command, producing
over 100 million quiet chess positions evaluated at search depth 9.
A "quiet" position is one with no captures, checks, or promotions
available on the next move — ensuring training labels are stable and
reliable. The dataset (large_gensfen_multipvdiff_100_d9.binpack) applies
multi-PV filtering: only positions where the top two candidate moves are
within 100 centipawns of each other are retained, ensuring genuinely
contested, non-trivial training examples. Data is stored in .binpack
format (~32 bytes/position), decoded at ~500,000–2,000,000 positions/sec
via a compiled C++ SparseBatchDataset loader.

STAGE 2 — FEATURE ENGINEERING (HalfKAv2_hm^):
Each chess board position is encoded as a sparse binary feature vector
using the HalfKAv2 (Half King-All version 2, Horizontally Mirrored,
Factorized) representation. The feature index for each piece is:

    f = k × (P × S) + p × S + s

where k ∈ [0,63] is the king square, P=10 piece types, S=64 squares,
p the piece type index, s the piece square. The total before mirroring
is 64 × 10 × 64 = 40,960 features per side; horizontal mirroring halves
this to 24,576 effective features per side. A typical chess position
activates only ~30 out of 24,576 features, enabling ultra-efficient
sparse accumulator updates.

STAGE 3 — DUAL NEURAL NETWORK ARCHITECTURE (NNUEModel via nnue-pytorch):
Two NNUE networks are trained using the official nnue-pytorch NNUEModel
class (training/model/model.py), configured via TrainingConfig:

  BigNet (output.nnue):
  - Feature Transformer: 24,576-dim sparse embedding → L1=256 accumulator
  - PSQT shortcut: 8 learned piece-square table outputs
  - Perspective-aware merge: concatenate White+Black accumulators (2×256)
  - Product Pooling (SqrCReLU): 512 → 256 non-linear features
  - 8 independent Layer Stacks (by piece count bucket):
      L1: FactorizedStackedLinear [256 → 31+1], SqrCReLU → 2×31
      L2: StackedLinear [62 → 32]
      L3: StackedLinear [32 → 1]
  - Total parameters: ~6.8M

  SmallNet (small_output.nnue):
  - Same architecture but L1=128 (vs 256 for BigNet)
  - L2=15, L3=32
  - Total parameters: ~3.5M

Key architectural innovations:
  - INCREMENTAL ACCUMULATOR: When a piece moves, only the rows for
    changed features are added/subtracted from the accumulator:
        a_new = a_prev − Σ(removed features) + Σ(added features)
    King moves trigger full accumulator refresh. This reduces per-position
    evaluation from O(N²) matrix multiplication to O(30) vector additions.
  - PRODUCT POOLING (SqrCReLU): Captures multiplicative cross-perspective
    interactions (e.g., pin/tactic involving pieces from both sides).
  - PSQT SHORTCUT: Piece-Square Table values are learned as a direct
    linear shortcut from the feature transformer, providing a fast
    material-counting baseline that accelerates convergence.
  - LAYER STACK BUCKETS: 8 independent sub-networks selected by piece
    count: bucket = min(⌊(n_pieces−1)/4⌋, 7). This allows phase-specific
    specialisation (opening/middlegame/endgame) within one binary.

STAGE 4 — TRAINING:
  Optimizer: Ranger21 (RAdam + Lookahead + Gradient Centralization)
  Loss: Symmetric Sigmoid Power Loss — converts centipawn scores to
    win-probability space via sigmoid, then applies |p_f - q_f|^2.5
    power penalty. Exponent 2.5 penalises large errors more than MSE
    while being lenient on small ones.
  Weight Clipping: |W_emb| ≤ 127/64=1.984, |W_out| ≤ (127²)/(600×16)
    — ensures float32 weights remain within int8/int16 representable
    range for loss-free quantization.
  Mixed Precision: bfloat16 on CUDA for memory efficiency.

  BigNet results: 400 epochs, 25M positions/epoch, RTX 3060 12GB
    Loss: 0.00542 → 0.00230 (57.6% reduction, ~147 hours total)
  SmallNet results: 75 epochs, 20M positions/epoch, RTX 4050 Laptop
    Loss: 0.00556 → 0.00276 (50.4% reduction, ~15 hours total)

STAGE 5 — QUANTIZATION AND BINARY EXPORT:
Float32 weights are quantized:
  - Feature Transformer weights: float32 → int16 (scale factor 64)
  - Dense layer weights: float32 → int8
  - Biases: int32
The .nnue binary layout:
  [4 bytes] Version/Magic header
  [N bytes] Feature transformer weights (int16)
  [N bytes] Feature transformer biases (int16)
  [N bytes] L1/L2/L3 weights (int8 × 8 buckets)
  [N bytes] L1/L2/L3 biases (int32 × 8 buckets)
Output sizes: BigNet ~6.5MB, SmallNet ~3.5MB

STAGE 6 — C++ ENGINE (SEARCH + INFERENCE):
The C++ engine (deepcastle binary, ~970KB) is derived from Stockfish 18's
source code (GPLv3). Key components:
  - search.cpp: Alpha-Beta + PVS (Principal Variation Search), Iterative
    Deepening, Aspiration Windows, Late Move Reductions (LMR), Null Move
    Pruning, Quiescence Search
  - evaluate.cpp: Adaptive dual-NNUE switching:
      * SmallNet used when |simple_eval| > 962 centipawns
      * If SmallNet score |v| < 277 cp, BigNet re-evaluated for accuracy
      * BigNet used for all balanced positions
  - nnue/nnue_accumulator.cpp: Incremental SIMD accumulator updates
  - movepick.cpp: Move ordering (hash move → MVV-LVA captures → killers
    → history heuristic)
  - tt.cpp: Transposition table (position hash → depth/score/best move)
  Performance: ~400,000–600,000 nodes/sec (NPS), evaluating each
  position in ~50 nanoseconds via SIMD (SSE4.1) integer arithmetic.
  Communicates via the standard UCI (Universal Chess Interface) protocol.

DUAL-NNUE SWITCHING LOGIC (from evaluate.cpp):
  bool use_smallnet = |simple_eval| > 962 centipawns
  if (smallNet && |nnue| < 277 cp) → re-evaluate with BigNet
  final_value = (125×psqt + 131×positional) / 128
  blended with: optimism * nnueComplexity / 476
  material-scaled: v = (nnue×(77871+material) + optimism×(7191+material)) / 77871

STAGE 7 — FULL-STACK WEB DEPLOYMENT:
  Backend: FastAPI (Python) UCI bridge deployed on Hugging Face Spaces
    via Docker. Features: singleton engine process, async I/O locking,
    background RAM cleanup, automatic hash clearing, WebSocket multiplayer,
    game analysis API with move classification (Brilliant/Great/Best/
    Excellent/Good/Inaccuracy/Mistake/Blunder), opening database lookup.
  Frontend: Next.js 16 (React 19) web app deployed on Vercel featuring:
    real-time evaluation bar, move analysis, game review, FEN-based
    position analysis, Chess960 variant support, P2P multiplayer via
    WebSocket, game history.

7. WHAT IS NOVEL ABOUT THE INVENTION?
────────────────────────────────────────────────────────────────────────

NOVELTIES:

1. FIRST END-TO-END DOCUMENTED DUAL-NNUE PIPELINE FROM SCRATCH:
   The complete pipeline — from quiet-position data generation →
   HalfKAv2 feature encoding → dual-network training → quantization-aware
   weight clipping → .nnue binary export → C++ integration → web deployment
   — is implemented, validated, and documented as a single cohesive system.
   Prior implementations exist only as fragmented components across
   different repositories without a unified, reproducible end-to-end system.

2. ADAPTIVE DUAL-NNUE EVALUATION WITH CUSTOM-TRAINED WEIGHTS:
   DeepCastle implements the SmallNet/BigNet adaptive switching scheme (a
   pattern introduced in Stockfish 16) but with independently trained
   custom weights for BOTH networks — not pre-trained weights. The
   switching logic (SmallNet when |simple_eval| > 962cp, re-evaluate
   with BigNet when |SmallNet score| < 277cp) is implemented with custom
   NNUE brains trained specifically for this system.

3. NOVEL TRAINING DIAGNOSTIC: SIGMA INITIALISATION FIX FOR BIAS-FREE EMBEDDINGS:
   A critical training fix was discovered: using σ=0.025 initialisation
   for the nn.Embedding layer (vs the nnue-pytorch default σ=0.007 designed
   for biased sparse linear layers). With bias-free embeddings, σ=0.007
   collapses the accumulator to near-zero, making product pooling output
   zero and producing a degenerate loss from the first batch. This fix
   enabled successful training convergence that failed without it.

4. CONSUMER-GPU DUAL-NNUE TRAINING WITH MINIMAL RESOURCES:
   BigNet (6.8M parameters, 400 epochs, 10B positions total) trained on
   a single NVIDIA RTX 3060 12GB GPU in ~147 hours. SmallNet (3.5M params,
   75 epochs) trained on an NVIDIA RTX 4050 Laptop GPU in ~15 hours.
   This demonstrates NNUE training is achievable without data center resources.

5. FULL-STACK CHESS AI AS A CLOUD-NATIVE WEB APPLICATION:
   The invention transforms a traditionally desktop-only chess engine into
   a cloud-native, browser-accessible application with real-time evaluation,
   game analysis with move classification, P2P multiplayer, Chess960
   variant support, and a sophisticated Next.js user interface — all on
   commodity cloud infrastructure.

6. SYMMETRIC SIGMOID POWER LOSS (|p_f − q_f|^2.5) APPLICATION:
   The invention applies the official Stockfish symmetric sigmoid power-law
   loss to a custom training scenario, converting centipawn scores to
   win-probability space before computing loss. The 2.5 exponent provides
   superior gradient behaviour compared to standard MSE for chess evaluation
   training, with empirical validation through convergence from 0.00542 to
   0.00230 over 400 epochs.

7. FACTORIZEDSTACKEDLINEAR WITH SHARED BUCKET COMPONENT:
   The layer stack design uses FactorizedStackedLinear at L1 — a module
   that merges bucket-specific weights with a shared "factorized" component
   that applies equally across all 8 buckets. This reduces bucket-specific
   overfitting while allowing phase-specific specialisation.

  - Faster to adopt: complete containerised deployment with single Dockerfile
  - More efficient: consumer-GPU trainable (~147h BigNet vs enterprise clusters)
  - Better documented: end-to-end pipeline documented from data to browser
  - Accessible: web browser interface — no local installation required
  - Novel training insight: sigma initialisation fix for stable convergence
  - Measured performance: 45% draw rate vs Stockfish 18 at blitz, ~3437 Elo

10. LIMITATIONS AND DISADVANTAGES OF THE INVENTION
────────────────────────────────────────────────────────────────────────

1. EVALUATION STRENGTH GAP:
   BigNet L1=256 vs Stockfish 18's L1=1024 means lower evaluation
   capacity. The initial −214.8 Elo gap vs SF18 is primarily due to
   this architectural difference and training data scale (100M vs billions).
   The extended retest (in progress) against strength-limited Stockfish
   (UCI_Elo=3190) is providing a more statistically robust sample (180
   games vs 20 games) with preliminary results showing DeepCastle is
   competitive at the ~3300–3355 Elo range.
   MITIGATION: Train BigNet with L1=1024 given sufficient GPU resources.

2. SINGLE-THREADED SEARCH:
   Current implementation uses 1 search thread. Stockfish 18 supports
   up to 1024 parallel threads for multi-core speedup.
   MITIGATION: Stockfish's multi-threading framework is present in the
   codebase and can be enabled by increasing the Threads UCI option.

3. TRAINING DATA SCALE:
   100M positions vs billions used by Stockfish team. More data would
   improve evaluation accuracy, especially in rare positions.
   MITIGATION: Generate additional data using gensfen command; use
   distributed training across multiple GPUs.

4. COLD START LATENCY:
   The first request to the Hugging Face Spaces backend starts the engine
   process (sub-second), but HF Spaces may sleep after inactivity.
   MITIGATION: Health-check pinging via GitHub Actions keeps the space warm.

5. CLOUD CPU NPS LIMITATION:
   Cloud CPUs yield ~400,000–600,000 NPS vs ~5,000,000+ on modern desktops.
   MITIGATION: Acceptable for web use; SSE4.1 SIMD compilation chosen for
   broad cloud CPU compatibility.

6. NO WDL BLENDING:
   Current training uses λ=1.0 (pure score labels). WDL blending (mixing
   score and win-draw-loss outcome labels) could improve calibration.
   MITIGATION: Planned for future training runs.

7. MEMORY USAGE ON SHARED INFRASTRUCTURE:
   HF Spaces memory is limited; engine hash table and model loading can
   spike above 2GB. Mitigated by background RAM cleanup task.

8. CHESS960 SERVER-SIDE EVALUATION:
   Chess960 starting FENs are generated client-side; the UCI engine
   may not correctly handle all Chess960 castling edge cases.
   MITIGATION: chess.js handles move validation client-side.

CAN LIMITATIONS BE OVERCOME?
  Yes — all limitations are resource/configuration constraints, not
  fundamental architectural issues. The core NNUE evaluation and training
  methodology is sound (validated by 45% draw rate vs SF18).

KEY INVENTIVE FEATURES WITH ATTRIBUTES/LIMITATIONS:

Feature 1: Dual-NNUE adaptive evaluation
  - SmallNet threshold: |simple_eval| > 962 centipawns
  - BigNet fallback threshold: |SmallNet score| < 277 centipawns
  - BigNet dimensions: L1=256, L2=31, L3=32 (6.8M parameters)
  - SmallNet dimensions: L1=128, L2=15, L3=32 (3.5M parameters)

Feature 2: HalfKAv2_hm^ feature encoding
  - 24,576 features per side (before serialisation)
  - 22,528 real features after serialisation (PS_NB = 11 × 64)
  - Typical active features per position: ~30 out of 24,576

Feature 3: Training pipeline
  - Training script: train.py (PyTorch Lightning + tyro CLI)
  - Serialization script: serialize.py (NNUEWriter, LEB128 compressed)
  - Dataset: depth-9, quiet-position, multipvdiff ≤ 100 centipawns
  - Dataset size: 100M+ positions
  - Loss function: |p_f − q_f|^2.5 (symmetric sigmoid power)
  - Loss scaling: NNUE2SCORE = 600.0; IN_SCALING = 340.0, OUT_SCALING = 380.0
  - Optimizer: Ranger21, LR = 8.75×10⁻⁴, StepLR γ = 0.992
  - Weight clipping: |W_emb| ≤ 127/64, |W_out| ≤ 127²/(600×16)
  - Checkpoint format: PyTorch Lightning .ckpt (then serialize.py → .nnue)

Feature 4: Quantization and Serialization
  - Tool: serialize.py with NNUEWriter (nnue-pytorch standard writer)
  - Feature transformer: float32 → int16 (scale factor 64, LEB128 compressed)
  - Dense layers: float32 → int8 (per-layer scale factors)
  - Binary naming: nn-<sha256[:12]>.nnue (content-hash reproducible)
  - Inference speed: ~50 nanoseconds/position (SSE4.1 SIMD)
  - Architecture hash embedded in .nnue header for validation

Feature 5: Layer stack bucketing
  - Number of buckets: 8
  - Bucket selection: bucket = min(⌊(piece_count − 1)/4⌋, 7)
  - Bucket 7 = opening (≥29 pieces), Bucket 0 = deep endgame (≤4 pieces)

Feature 6: PSQT shortcut
  - 8 PSQT outputs (one per bucket) from feature transformer directly
  - PSQT contribution: (e^W_P − e^B_P) × (us − 0.5)
